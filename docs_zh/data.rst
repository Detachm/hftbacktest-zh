数据
====

关于行情数据的采集和转换，请参考 `Data Collector <https://github.com/nkaz001/hftbacktest/tree/master/collector>`_ 或官方教程 `Data Preparation <https://hftbacktest.readthedocs.io/en/latest/tutorials/Data%20Preparation.html>`_。

格式
----

``hftbacktest`` 可以读取 ``numpy`` 结构化数组。数据按下面顺序包含 8 个字段。更多细节可以参考 Rust 文档中的 `Event <https://docs.rs/hftbacktest/0.3.1/hftbacktest/types/struct.Event.html>`_。

* ev (u64)：事件标志。可用的标志组合见 `Constants <https://docs.rs/hftbacktest/0.3.1/hftbacktest/types/index.html#constants>`_。
* exch_ts (i64)：交易所时间戳，即事件在交易所发生的时间。
* local_ts (i64)：本地时间戳，即本地接收到该事件的时间。
* px (f64)：价格。
* qty (f64)：数量。
* order_id (u64)：订单 ID，仅用于 L3 Market-By-Order 行情。
* ival (i64)：预留的额外 i64 字段。
* faval (f64)：预留的额外 f64 字段。

校验
----

1. 所有时间戳都必须按正确的时间顺序排列。

某些情况下，一个事件在交易所发生得更早，因此交易所时间戳更早，但本地接收时间却晚于另一个事件。这会导致交易所时间戳和本地时间戳的顺序相反。为处理这种情况，hftbacktest 使用 :const:`EXCH_EVENT <hftbacktest.types.EXCH_EVENT>` 和 :const:`LOCAL_EVENT <hftbacktest.types.LOCAL_EVENT>` 标志。

带有 :const:`EXCH_EVENT <hftbacktest.types.EXCH_EVENT>` 标志的事件，应按交易所时间戳保持时间顺序；带有 :const:`LOCAL_EVENT <hftbacktest.types.LOCAL_EVENT>` 标志的事件，应按本地时间戳保持时间顺序。

2. 交易所时间戳必须早于本地时间戳；行情延迟必须为正。

由于两个站点之间可能存在时间同步误差，本地时间戳可能早于交易所时间戳，从而产生负延迟。最好的解决方式是使用 PTP (Precision Time Protocol) 改善时间同步，尽量降低负延迟出现的概率。

不过，也可以通过增加基础延迟，或按负延迟的幅度整体平移时间戳，来保证数据只包含正延迟，即事件的本地时间戳总是晚于交易所时间戳。
